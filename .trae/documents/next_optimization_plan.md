# 下一步优化计划建议

根据目前项目的代码结构（特别是 `dice-tales` 前端部分），我整理了三个主要的优化方向。这些优化能够提升项目的稳定性、可维护性以及玩家的沉浸感。

## 方案一：底层架构与代码重构（推荐优先进行）

**1. 引入 Zustand 重构状态管理**
*   **原因解释**：目前游戏的数据（如角色、冒险进度）使用的是简单的 `useState` 结合 `localStorage` 存放。这意味着如果页面里有多个独立模块同时读取数据，它们之间的数据可能无法实时同步。引入 Zustand 可以提供真正的“全局状态中心”，并自带更稳健的本地存储功能，确保玩家切换页面或进行操作时数据绝对一致。

**2. 拆分超大组件**
*   **原因解释**：当前的游戏主界面（`app/game/[adventureId]/page.tsx`）包含了近千行代码，将聊天面板（`ChatPanel`）、地图面板（`MapPanel`）都塞在了一起。将它们拆分成独立的文件，就像把杂乱的工具箱分类整理好，未来添加新功能或修复 Bug 时会更加清晰高效，不容易“牵一发而动全身”。

**3. 环境变量配置后端 API**
*   **原因解释**：目前代码里直接写死了后端的地址 `http://127.0.0.1:8000`。这就像把家里的门牌号焊死在了车上，未来如果我们想把游戏发布到网上，就必须手动去改每一处代码。提取为环境变量后，系统会自动根据是“本地开发”还是“线上运行”来切换地址。

## 方案二：数据结构与潜在隐患修复

**1. 统一角色背包的数据字段**
*   **原因解释**：在代码检查中发现，有些地方角色背包使用的是 `items` 字段，有些地方又更新为了 `inventory` 字段。这种不一致就像是同一个物品有了两个不同的名字，在未来读取背包或装备物品时，极易导致系统找不到数据而报错（比如游戏卡死）。需要全局统一数据结构。

## 方案三：跑团体验与界面打磨

**1. 增加“AI 思考中”的视觉反馈**
*   **原因解释**：当玩家发送行动后，AI DM 生成回复可能需要几秒钟。目前只有简单的状态锁定，如果能加入“DM 正在翻阅规则书...”或者骰子转动的加载动画，能极大地缓解玩家的等待焦虑，提升沉浸感。

**2. 完善断网/后端无响应的错误处理**
*   **原因解释**：目前如果后端 Python 服务没开，前端会直接在聊天框抛出系统错误。我们可以优化这里的体验，提供一个更友好的重连按钮或弹窗提示。
