# 后台管理端与模组结构增强实施计划

## 1. 目标与验收标准
- 目标1：提供可用的后台管理端，支持查看游戏数据、用户数据、导入模组、编辑 AI 解析结果。
- 目标2：扩展结构化模组数据模型，补齐剧情顺序、触发条件、线索、手outs/资料、场景物件、图片资源等缺失信息。
- 目标3：让扩展后的结构化信息可被 DM 流程消费（至少可查询、可触发、可发放给玩家并进入记录/物品栏）。
- 验收标准：
  - 存在 `/admin` 入口及至少 4 个可用子页：数据总览、用户管理、模组导入、模组编辑。
  - 管理端完成基础鉴权保护，未登录用户不可访问。
  - 可导入 DOCX/PDF 模组并生成结构化结果，支持人工修订后保存。
  - 结构化模组 API 可读写新增字段，且旧数据可兼容读取。
  - 游戏流程可读取至少一类新增触发内容（如线索/手outs）并完成发放记录。

## 2. 范围与边界
- 本次纳入：
  - 管理端 UI（Next.js App Router）与后台 API（FastAPI + Next 代理）联动。
  - 结构化 schema 升级与导入链路升级（FastAPI prompt + Dify Workflow schema/prompt）。
  - 基础数据一致性治理（优先统一管理端走 FastAPI 数据源）。
- 本次不纳入：
  - 复杂 RBAC（仅先做管理员级别访问控制）。
  - 完整可视化流程编辑器（先做表单 + JSON 混合编辑）。
  - 图片自动 OCR / 图像语义理解（先支持上传与元数据关联）。

## 3. 现状问题拆解
- 管理端仅有上传页，缺少统一后台入口、数据查询和编辑闭环。
- 结构化模型只覆盖 title/background/locations/npcs/events/quests，缺少流程控制与素材资产字段。
- 触发逻辑仅文本化，无法可靠驱动“满足条件后给玩家线索/资料/物品”。
- 新导入模组与前端模组展示存在链路断层（静态列表与动态数据并存）。
- 存在数据面分裂（FastAPI 与 Supabase 并行），后台管理应先固定单一来源避免认知混乱。

## 4. 目标架构与实现策略
- 管理端前端：在 `app/admin/*` 下建立模块化页面；所有管理端请求统一走 `/api/backend/*` 代理到 FastAPI。
- 管理端后端：复用并补充 FastAPI 管理 API，避免直接在管理端写 Supabase 逻辑。
- 结构化模型：扩展 `StructuredModule` 及子对象，采用“向后兼容 + 渐进增强”策略。
- 解析链路：同步升级两处定义源（FastAPI prompt + Dify workflow prompt/schema），保证字段一致。
- 运行消费：在 DM 动作环节最小接入触发器评估与奖励发放，先打通 1 条可验证链路。

## 5. 详细实施步骤

### 步骤A：后台管理端框架搭建
- 新增 `/admin/page` 作为后台首页，展示四类入口卡片：游戏数据、用户数据、模组导入、模组编辑。
- 新增 `/admin/layout` 与导航（Tabs 或 Sidebar），形成统一后台壳层。
- 保留并重构现有 `/admin/upload`，将其并入后台导航，不再孤立存在。

### 步骤B：基础鉴权与安全收敛
- 增加管理端访问保护（middleware 或服务端校验），以环境变量维护管理员口令或令牌。
- 移除前端硬编码敏感信息（例如 API Key 默认值），改为后端读取环境变量。
- 管理端所有写操作接口增加最小鉴权校验与错误提示规范（401/403/422）。

### 步骤C：管理 API 补齐（FastAPI）
- 新增后台查询接口：
  - `GET /admin/stats`：汇总模块数、角色数、会话数、最近活跃时间。
  - `GET /admin/users`：用户列表与关联统计（角色数、最近会话）。
  - `GET /admin/sessions`：会话列表（按用户/模组/时间过滤）。
- 新增模组管理接口：
  - `GET /admin/modules`：分页/搜索模组（含是否有 structured）。
  - `GET /admin/modules/{id}`：获取结构化详情。
  - `PUT /admin/modules/{id}/structured`：保存编辑结果（带 schema 校验）。
  - `POST /admin/modules/{id}/assets`：上传图片/附件并建立引用。
- 统一返回格式（data + meta + error）以便前端表格与分页复用。

### 步骤D：结构化模组 Schema 扩展
- 扩展 `StructuredModule` 顶层字段：
  - `sequence`: 场景/章节顺序数组。
  - `triggers`: 结构化触发规则数组（条件、动作、冷却/一次性）。
  - `clues`: 线索池（来源、发现条件、可见对象）。
  - `handouts`: 可发放资料（文本/图片/混合、发放条件、入背包策略）。
  - `scene_items`: 场景物件（可交互动作、状态变化、关联线索）。
  - `assets`: 图片/地图/附件资源元数据。
- 扩展子模型并定义最小必填字段、枚举值与版本号（如 `schema_version`）。
- 兼容策略：旧记录读取时自动填充默认空数组；写入时按新 schema 校验。

### 步骤E：导入链路升级（AI 解析）
- 更新 FastAPI `ai_structure_module` 提示词，使其显式抽取新增字段。
- 更新 `dify_workflow/模组提取.yml` 的 prompt 与 structured_output schema，保证与后端模型一致。
- 为“无法自动抽取”的字段设置 `confidence` 或 `notes`，便于后台人工补完。
- 导入流程改造为“两段式”：先生成草稿 structured，再进入后台编辑发布。

### 步骤F：后台模组编辑器
- 新增 `/admin/modules` 列表页：搜索、筛选、进入编辑。
- 新增 `/admin/modules/[moduleId]/edit` 编辑页：
  - 左侧结构树（章节/场景/触发器/线索/手outs/资产）。
  - 右侧表单编辑（常规字段）+ JSON 高级编辑（复杂字段）。
  - 变更对比与保存校验提示（字段缺失、枚举不合法、引用断链）。
- 提供“预览 DM 触发结果”面板：模拟触发条件命中后会发放哪些内容。

### 步骤G：游戏消费链路最小打通
- 在会话状态中加入已发现线索、已发放资料、已触发规则记录。
- GM 动作处理中接入触发评估：
  - 根据当前场景、玩家动作标签、关键检定结果判断触发。
  - 命中后将线索/手outs写入会话状态与玩家可见内容。
- 前端游戏页至少展示“新获得线索/资料”提示与可查看入口。

### 步骤H：数据源统一与回归
- 管理端明确只对 FastAPI 数据源负责，避免与 Supabase 管理逻辑混用。
- 模组列表页改为支持动态来源（优先后端），减少静态常量依赖。
- 补充必要的数据迁移脚本：将现有 structured 模组补齐新字段默认值。

## 6. 测试与验证方案
- 后端：
  - 为新增 schema 与接口编写单元测试（校验成功/失败、兼容旧数据、触发命中逻辑）。
  - 为导入链路编写集成测试（DOCX/PDF -> structured 草稿 -> 保存）。
- 前端：
  - 管理端关键页面进行交互回归（列表、搜索、编辑、保存、错误提示）。
  - 验证鉴权拦截与未授权访问行为。
- 端到端：
  - 导入《鬼屋》文档，人工补充触发与线索后发布。
  - 启动新会话，验证满足条件后 DM 可发放线索/资料并在玩家侧可见。

## 7. 交付顺序（建议执行节奏）
- 第一批：步骤A + B + C（先形成安全可用后台骨架与数据可见性）。
- 第二批：步骤D + E + F（建立“可导入 + 可编辑”的主业务闭环）。
- 第三批：步骤G + H（接入游戏消费与数据统一，完成整体闭环）。

## 8. 风险与应对
- 风险1：AI 提取不稳定导致字段缺漏。
  - 应对：后台编辑器作为强制校对环节，保存前校验必填项。
- 风险2：旧数据结构不一致。
  - 应对：读取时兼容默认值 + 一次性迁移脚本。
- 风险3：触发规则过于复杂导致实现膨胀。
  - 应对：首版仅支持有限条件类型（场景、动作标签、检定阈值），后续迭代扩展。
- 风险4：双数据源继续引发歧义。
  - 应对：在管理端层面先锁定 FastAPI，后续再做全局数据层整合。

## 9. 本计划执行时的关键实现约束
- 所有管理端请求必须走同源代理 `/api/backend/*`。
- 不在前端代码中存放任何密钥或默认私密凭据。
- 新增字段必须提供后向兼容，不破坏现有会话读取。
- 每个新增 API 都应给出失败态与可读错误信息，便于运营排错。
