# 产品需求文档（PRD） - AI Dungeon Master 游戏网站

## 1. 项目概述
本项目是一个单人跑团文游网站，AI作为Dungeon Master（DM）为玩家生成基于 **Dungeons & Dragons**（DND）和 **Call of Cthulhu**（Coc）规则的冒险情节。玩家通过选择角色卡和模组开始游戏，AI通过文本描述、技能检定、战斗系统等方式驱动游戏进程。

### 集成的服务
- **Supabase**：用作后台数据库服务，存储玩家数据、游戏进度、模组信息等。使用 Supabase 的实时数据库和认证系统来管理用户信息和游戏状态。
- **OpenRouter**：提供 AI 生成服务，支持通过 API 调用生成自然语言内容，为 AI DM 生成文本描述、对话、任务等。

---

## 2. 核心功能

### 2.1 角色创建
- 玩家选择游戏规则（DND 或 Coc）。
- 输入角色的基本信息（如名字、背景、技能、职业、种族等）。
  - **DND**：选择种族（如人类、精灵等）和职业（如战士、巫师等）。
  - **Coc**：选择背景并生成属性（如力量、体质、智力等）。

### 2.2 游戏选择与模组加载
- 玩家选择游戏模组（如《The Haunting》）。
- 加载相关情节和规则，并生成 AI DM 响应。

### 2.3 游戏界面
- **聊天框**：显示 AI DM 的文本描述和玩家输入。
- **角色属性栏**：显示玩家角色的生命值、技能、任务进度等。
- **物品栏**：显示并允许玩家使用物品。
- **任务日志**：记录当前任务及已完成任务。
- **地图（可选）**：展示玩家当前的位置和游戏场景。

### 2.4 游戏互动与战斗
- 玩家可以进行调查、战斗、与NPC对话等操作。
- 战斗根据玩家角色和敌人的属性进行掷骰判定，并展示战斗结果。
- 玩家与 AI DM 的互动决定游戏的进展，AI DM 会基于模组规则和玩家选择推动情节发展。

---

## 3. 数据流与交互

### 3.1 前端与后端的交互
- 玩家每次操作后，前端发送请求到后端，后端根据游戏规则和模组生成结果，返回给前端更新 UI。
  - 例如：玩家选择“调查书桌”，后端处理该请求，计算是否能找到线索，返回结果并展示。

### 3.2 后端功能与 AI DM
- 后端生成 AI DM 的回答和后续情节（如战斗结果、任务更新等），返回给前端展示。

---

## 4. 需要生成的代码功能

### 4.1 前端功能

1. **角色创建页面**：
   - **选择 DND/Coc**：两个选项，玩家选择后根据规则生成角色属性。
   - **生成角色卡**：输入角色信息（名字、背景、技能等），生成并展示角色卡。

2. **游戏主界面**：
   - **聊天框**：动态显示 AI DM 的描述与玩家输入。
   - **角色信息栏**：显示玩家角色的当前属性（例如生命值、理智值等）。
   - **任务日志**：记录并展示当前任务。
   - **物品栏**：展示当前物品，可以进行点击操作。
   - **选择按钮**：当 AI 提供选项时，生成多个选择按钮，玩家点击后更新游戏情节。

3. **战斗界面**：
   - **战斗按钮**：攻击、防御、技能等按钮，触发战斗操作。
   - **战斗结果显示**：战斗后显示角色和敌人的健康变化、战斗结果。

### 4.2 后端功能

1. **角色创建与管理**：
   - 接收角色创建数据（如背景、职业、技能）并生成角色数据。
   - 存储角色属性（DND 的种族、职业，Coc 的属性和理智等）。

2. **游戏模组管理**：
   - 加载游戏模组数据，解析模组中的事件、NPC、任务等。
   - AI DM 根据模组和玩家行为生成响应。

3. **AI DM 引擎**：
   - 根据模组规则生成情节、战斗、任务等。
   - 处理玩家的技能检定、战斗、事件触发等。

4. **战斗与技能处理**：
   - 根据玩家和敌人的属性计算战斗结果（如攻击、伤害、回合顺序等）。
   - 根据 DND 和 Coc 的规则执行技能检定、理智检定等。

### 4.3 数据库设计

- **玩家数据**：
  - 存储玩家角色的属性、技能、物品、任务进度等信息。
- **模组数据**：
  - 存储模组的事件链、NPC 对话、任务目标、战斗规则等。
- **游戏状态**：
  - 存储当前的游戏进度，包括已完成的任务、AI DM 的反馈、战斗状态等。

---

## 5. AI DM 逻辑需求

### 5.1 模组解析
- 从 JSON/YAML 格式的模组数据中加载并解析任务、事件、NPC、战斗规则。

### 5.2 情节生成
- 根据玩家选择、任务进度和规则生成下一步情节。
  - 例如，玩家调查某物品后，AI DM 根据模组规则给出提示或新任务。

### 5.3 战斗与技能处理
- AI DM 根据模组中的战斗规则和玩家行为计算战斗结果，并给出描述。

---

## 6. 生成代码的 API 需求

1. **角色创建接口**：
   - **POST** `/api/create-character`: 接收玩家输入的角色信息，生成并返回角色数据。
   
2. **游戏开始接口**：
   - **POST** `/api/start-game`: 接收玩家选择的模组，加载模组数据并初始化游戏状态。

3. **玩家行为接口**：
   - **POST** `/api/player-action`: 接收玩家输入的行为（如“调查”或“攻击”），返回 AI DM 生成的情节或战斗结果。

4. **战斗与技能检定接口**：
   - **POST** `/api/combat`: 处理战斗，计算伤害和结果。
   - **POST** `/api/skill-check`: 执行技能检定并返回结果。

## 8. 数据流与实时同步

### 8.1 游戏流程
1. **玩家选择游戏规则**（DND 或 Coc）并创建角色。
2. **AI DM 加载模组**并通过 OpenRouter 生成初始情节（例如，描述当前环境、任务目标等）。
3. **玩家操作**：通过点击按钮（如“调查”或“战斗”）与 AI DM 互动，AI DM 根据玩家的选择生成文本反馈，并更新任务日志、角色状态等。
4. **实时同步**：每次游戏进展（如任务完成、技能检定、战斗结束等）都会通过 Supabase 实时数据库更新，确保前端实时显示最新信息。

### 8.2 示例请求与响应
- **玩家选择“调查房间”**：
  - **请求**：`POST /api/player-action`，请求体包含玩家选择：“调查房间”。
  - **响应**：后端调用 OpenRouter 生成描述，“你走近桌子，发现了一本陈旧的书籍”。返回给前端显示。

- **玩家参与战斗**：
  - **请求**：`POST /api/combat`，请求体包含战斗相关数据（如玩家角色和敌人信息）。
  - **响应**：后端根据战斗规则计算结果，返回战斗描述和伤害计算结果，“你成功击中敌人，造成了10点伤害”。

## 9. 安全性与认证
- 使用 **Supabase** 的身份验证系统，确保玩家数据的安全和隐私。
- 使用 **JWT**（JSON Web Tokens）来保护 API 接口，确保只有经过验证的用户才能执行特定操作。

---

## 10. 数据库设计（Supabase）

### 10.1 数据库结构
- **玩家数据表**：
  - 存储玩家角色信息（例如名字、背景、技能、当前状态等）。
- **任务表**：
  - 存储任务数据、进度、相关事件。
- **游戏日志**：
  - 存储游戏过程中生成的事件、AI DM 的描述等信息。

### 10.2 实时数据库
- 使用 **Supabase 的实时功能**，每当游戏数据（如玩家状态、任务进度）发生变化时，前端可以实时接收到更新。

---

## 11. 开发与部署
- **开发环境**：使用 Docker 容器化部署后端服务，确保本地开发与生产环境的一致性。
- **部署**：将后端部署在 Vercel、Heroku、或 AWS 等云平台，前端可以部署在 Netlify 或 Vercel。
- **CI/CD**：使用 GitHub Actions 或 GitLab CI/CD 进行持续集成与部署，确保代码更新后能够自动部署。

---

## **总结**
- **Supabase**：用于存储玩家数据、游戏进度和任务信息，同时提供实时同步功能。
- **OpenRouter**：通过调用 API 生成 AI DM 的自然语言内容，为玩家提供动态的游戏描述、任务和对话。
- **WebSocket**：确保前后端交互的实时性，支持玩家操作后的即时反馈。
